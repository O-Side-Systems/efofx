Refactor tenant_service.py to close Gap 1 from the verification report. Three methods need fixing:

**1. get_tenant_statistics(tenant_id)** (lines 135-175):
- REMOVE: `get_estimates_collection()` and `get_feedback_collection()` calls (these are not imported and would NameError)
- REMOVE: All `ObjectId(tenant_id)` calls — tenant_id is a UUID string now
- REMOVE: `from bson import ObjectId` local import
- ADD: Import `get_tenant_collection` from `app.db.mongodb` and `DB_COLLECTIONS` from `app.core.constants`
- USE: `estimates_col = get_tenant_collection(DB_COLLECTIONS["ESTIMATES"], tenant_id)` for estimates queries
- USE: `feedback_col = get_tenant_collection(DB_COLLECTIONS["FEEDBACK"], tenant_id)` for feedback queries
- The TenantAwareCollection auto-injects tenant_id, so remove `"tenant_id": ObjectId(tenant_id)` from all filters and aggregation `$match` stages — just use `{}` as the filter (TenantAwareCollection handles scoping)
- For the aggregation pipeline in feedback avg_rating: remove `{"$match": {"tenant_id": ObjectId(tenant_id)}}` — TenantAwareCollection.aggregate() prepends a $match stage automatically

**2. validate_tenant_limits(tenant_id)** (lines 177-211):
- Same pattern: replace `get_estimates_collection()` with `get_tenant_collection(DB_COLLECTIONS["ESTIMATES"], tenant_id)`
- Remove `ObjectId(tenant_id)` from filter — use `{"created_at": {"$gte": start_of_month}}` only (TenantAwareCollection adds tenant_id)
- Change `self.get_tenant(tenant_id)` to `self.get_by_tenant_id(tenant_id)` — get_tenant uses ObjectId lookup (legacy), get_by_tenant_id uses UUID string
- Adapt the tenant limit check: get_by_tenant_id returns a dict, not a Tenant model — use `.get("settings", {}).get("max_estimations_per_month", 1000)` or a sensible default if the field is absent (the field `max_estimations_per_month` may not exist on the new Tenant model — check the model and use tier-based defaults if needed: trial=100, paid=1000)

**3. get_all_tenant_statistics()** (lines 213-238):
- This is a cross-tenant admin aggregation — it intentionally counts across ALL tenants
- Do NOT use TenantAwareCollection here (it scopes to one tenant)
- USE: `get_collection(DB_COLLECTIONS["ESTIMATES"])` and `get_collection(DB_COLLECTIONS["FEEDBACK"])` for raw unscoped access
- Add import of `get_collection` from `app.db.mongodb`
- These raw calls are correct because this is platform-level admin stats, not tenant-scoped data

**Top-level import changes:**
- Change: `from app.db.mongodb import get_tenants_collection` to `from app.db.mongodb import get_tenants_collection, get_tenant_collection, get_collection`
- Add: `from app.core.constants import DB_COLLECTIONS`
- Remove the `from bson import ObjectId` local imports inside the three methods (they are no longer needed)

**Also fix get_tenant() and update_tenant() and deactivate_tenant():**
- These use `ObjectId(tenant_id)` for `_id` lookup (legacy Mongo ObjectId). They should use `{"tenant_id": tenant_id}` filter on the tenants collection instead, since the new design uses UUID string `tenant_id` field. But since these operate on the tenants collection (not tenant-scoped), keep using `get_tenants_collection()` (which is NOT deprecated — it's the correct raw accessor for the top-level tenants collection).
- Replace `{"_id": ObjectId(tenant_id)}` with `{"tenant_id": tenant_id}` in get_tenant, update_tenant, deactivate_tenant
- Remove the `from bson import ObjectId` local imports from these methods

**Do NOT touch:** get_by_tenant_id, get_by_email, get_tenant_by_api_key, list_tenants, create_tenant — these already work correctly.
