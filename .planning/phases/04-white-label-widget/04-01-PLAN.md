**Step 4: Create `src/components/ErrorBoundary.tsx`** — wraps the entire widget.

Use `react-error-boundary` (just installed). The component:

- Export as `WidgetErrorBoundary`

```typescript
import { ErrorBoundary } from 'react-error-boundary';

export function WidgetErrorBoundary({ children }: { children: React.ReactNode }) {
  return (
    <ErrorBoundary
      fallback={null}
      onError={(error, info) => {
        console.error('[efOfX widget] Runtime error:', error, info);
      }}
    >
      {children}
    </ErrorBoundary>
  );
}
```

**Step 2: Modify `src/components/ShadowDOMWrapper.tsx`** — inject CSS into shadow root.

Replace the existing implementation. Key changes:
1. Import widget CSS as string: `import widgetStyles from '../widget.css?inline';`
2. After `attachShadow({ mode: 'open' })`, create a `<style>` element, set `textContent = widgetStyles`, append to shadow root BEFORE the container div.
3. Accept optional `branding?: BrandingConfig` prop. When branding is provided, create an additional `<style>` element that overrides `:host` custom properties:
   ```
   :host { --brand-primary: ${branding.primary_color}; ... }
   ```
4. Keep `mode: 'open'` (not closed — per research, closed mode breaks debugging).

Critical: The `vite-plugin-css-injected-by-js` injects CSS into `document.head` which does NOT reach Shadow DOM. The `?inline` import returns the CSS as a string for manual injection. This is the ONLY way widget styles work in Shadow DOM.

**Step 6: Replace `src/main.tsx`** — IIFE entry point.

Replace the existing placeholder entry point. Critical implementation details:

1. **Read data attributes synchronously** at module top level (before any async operations). `document.currentScript` is ONLY available during synchronous execution:
   ```typescript
   const _scriptEl = document.currentScript as HTMLScriptElement | null;
   ```

2. `getScriptConfig()` function reads from `_scriptEl` (stored above) or falls back to `document.querySelector('script[data-api-key]')`:
   - `data-api-key` → `config.apiKey`
   - `data-mode` → `config.mode` (default: `'floating'`)
   - `data-container` → `config.containerId` (default: `'efofx-widget'`)

3. `init(config?)` function:
   - Wraps EVERYTHING in try/catch — on error, `console.error('[efOfX widget] initialization failed', e)` and return `{ destroy: () => {} }` (WSEC-01)
   - Merges script config with passed config
   - For floating mode: create a new div, append to `document.body`, set `style.zIndex = '2147483647'`
   - For inline mode: find container by `containerId`, fall back to creating one
   - `createRoot(container).render(...)` with `WidgetErrorBoundary > ShadowDOMWrapper > App`
   - Return `{ destroy: () => root.unmount() }`

4. Auto-init in dev mode: `if (import.meta.env.DEV) { init(); }`

5. IIFE export: `export default { init };`

**Step 7: Update `vite.config.ts`** — ensure CSS is NOT injected into document.head for production build.

The `cssInjectedByJsPlugin()` currently injects CSS into `document.head`. For the widget, ALL CSS goes into Shadow DOM via `?inline`. Keep the plugin for dev mode but ensure the production IIFE build does not inject CSS into the host page. Add to build config:
```typescript

```
And configure `cssInjectedByJsPlugin` to work correctly with the Shadow DOM pattern. If the plugin still injects into `document.head` in production, disable it for production build and rely solely on `?inline` imports.
