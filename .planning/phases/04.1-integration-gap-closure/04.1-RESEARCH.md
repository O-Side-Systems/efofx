## Summary

Phase 4.1 is a targeted cleanup phase with four independent, well-scoped changes to existing backend Python code. All changes are in the `apps/efofx-estimate` FastAPI application. No new functionality is introduced — only bugs are fixed, startup wiring is corrected, and dead code is deleted.

The biggest risk is missing a call site. The `tenant.id` attribute does not exist on the `Tenant` model (which uses `tenant_id`), so any execution of the legacy `start_estimation`, `get_estimation`, `upload_image`, `_process_estimation`, and `submit_feedback`/related methods will raise `AttributeError`. These are the only affected call sites — the widget/chat/SSE primary path already uses `tenant.tenant_id` correctly. A grep-based audit confirms exactly 11 `tenant.id` references across two files, all in legacy paths.

The index startup gap (INT-02) requires a one-line addition to `main.py`'s lifespan and a decision about failure behavior — the user decided the app must fail to start if `create_indexes()` raises. This aligns with how `PromptService.load_all()` already behaves (raises on failure). The `create_indexes()` function already exists with full implementation in `mongodb.py`; it just needs to be called. The function is idempotent (MongoDB `create_index` is a no-op if the index already exists).

**Primary recommendation:** Execute four independent surgical changes in a single plan wave: (1) fix `tenant.id` → `tenant.tenant_id` in two service files, (2) wire `create_indexes()` in `main.py` lifespan, (3) remove POST `/estimate/start` route and its handler methods, (4) delete `rotate_openai_key()` and `get_tenant_allowed_origins()` with full trace cleanup.

### Pattern 2: Startup Index Wiring

**What:** `create_indexes()` is a fully implemented async function in `app/db/mongodb.py` that creates all compound tenant-scoped indexes (ISOL-03). It is never called. It must be called in `main.py`'s lifespan immediately after `connect_to_mongo()`.

**Current lifespan (main.py, lines 31-58):**
```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    logger.info("Starting efOfX Estimation Service...")
    try:
        await connect_to_mongo()
        logger.info("MongoDB connection established")
    except Exception as e:
        logger.error(f"Failed to connect to MongoDB: {e}")
    # ... PromptService.load_all() with raise on failure ...
    yield
```

**Required change:** Add `create_indexes()` call immediately after `connect_to_mongo()`. Per the user's decision, failure must raise (crash the app). The import of `create_indexes` must also be added to the `mongodb` import line.

```python
from app.db.mongodb import connect_to_mongo, close_mongo_connection, health_check as db_health_check, create_indexes

@asynccontextmanager
async def lifespan(app: FastAPI):
    logger.info("Starting efOfX Estimation Service...")
    await connect_to_mongo()
    logger.info("MongoDB connection established")
    await create_indexes()
    logger.info("Database indexes created")
    # PromptService.load_all() follows ...
```

**Key detail:** The existing `try/except` around `connect_to_mongo()` swallows the exception (logs and continues). Per user decision, `create_indexes()` must fail loudly. The cleanest approach is to let both calls raise outside the try/except, mirroring the `PromptService.load_all()` pattern that already raises. The try/except around `connect_to_mongo()` should be restructured so the exception propagates.

**`create_indexes()` is idempotent:** MongoDB's `create_index` is a no-op when the index already exists. Calling on every startup is safe.

**Additional logging requested:** The user asked for logging of each index name created or confirmed. The current `create_indexes()` only logs one final "Database indexes created successfully" message. Add per-index logging:
```python
await db["estimates"].create_index([("tenant_id", 1), ("created_at", -1)])
logger.info("Index confirmed: estimates.tenant_id_1_created_at_-1")
```

## Common Pitfalls

### Pitfall 1: Restructuring connect_to_mongo Exception Handling

**What goes wrong:** The current `try/except Exception` around `connect_to_mongo()` logs the error and continues. If `create_indexes()` is placed inside that try/except, an index failure will be swallowed.

**Why it happens:** The existing try/except was written to be permissive (perhaps for dev convenience). The user decision is now fail-fast.

**How to avoid:** Move `connect_to_mongo()` and `create_indexes()` outside the try/except, or let them raise. Match the `PromptService.load_all()` pattern: no catch, exception propagates to FastAPI/uvicorn which refuses to start.

**Warning signs:** If you see a `try/except` wrapping the new `create_indexes()` call that doesn't re-raise, it's wrong.

### Pitfall 2: Missing the EstimationResult Import Cleanup

**What goes wrong:** After removing `_parse_estimation_response()` and `_create_default_estimation()`, the `EstimationResult` import in `estimation_service.py` becomes unused. Leaving it is noise.

**Why it happens:** Import cleanup is easy to forget when focused on function deletion.

**How to avoid:** After removing methods, re-read the import block and check each imported name against what remains.

### Pitfall 3: Removing _classify_project Without Checking generate_from_chat

**What goes wrong:** `_classify_project()` is called by BOTH `_process_estimation()` (legacy, being removed) AND `generate_from_chat()` (active, kept). Removing `_classify_project()` would break the active SSE path.

**Why it happens:** It looks like "legacy code" because it's listed under the private methods associated with `start_estimation`.

**How to avoid:** `_classify_project()` must NOT be removed. It is used by `generate_from_chat()` at line 327. Only remove methods that are exclusively called from `start_estimation` or `_process_estimation`.

**Methods to KEEP (called by generate_from_chat):**
- `_classify_project()` — called at line 327
- `_build_description_from_context()` — called at line 319

**Methods to REMOVE (called only from start_estimation/_process_estimation):**
- `start_estimation()` — the legacy entry point itself
- `_process_estimation()` — called only by `start_estimation`
- `_generate_estimation()` — called only by `_process_estimation`
- `_parse_estimation_response()` — called only by `_generate_estimation`
- `_create_default_estimation()` — called only by `_generate_estimation`

### Pitfall 4: Removing EstimationResult from models import in routes.py

**What goes wrong:** `EstimationResult` is not imported in `routes.py` — it's in `estimation_service.py`. Checking the wrong file.

**Why it happens:** Both files are being edited simultaneously; easy to confuse which import belongs where.

**How to avoid:** After changes, re-read the import section of each edited file specifically.
